# ILang V2 — Documentation

Technical reference for the ILang format, CLI, and prompt pipeline.  
For a short getting-started guide, see [README.md](./README.md).

---

## 1. What ILang is

ILang is a **spec → prompt** toolchain:

1. You describe a coding task in a structured `.ilang` file (plain language).
2. The CLI validates sections and builds a compressed `prompt:` payload.
3. You paste **only** that payload into an AI chat.
4. The model is instructed to emit code under `external/` with minimal prose.

Primary goal: reduce **output tokens** (explanations, markdown, chatter).  
Secondary: keep tasks structured and language-agnostic via `lang:`.

ILang does **not** call an LLM API by itself. It prepares the prompt; you paste it into Cursor or another chat.

---

## 2. Architecture

```
internal/index.ilang     human sections + generated prompt:
        │
        ▼
   parse → check → dictionary → compress → rules
        │
        ▼
   prompt: { rules, dict, body }
        │
        ▼
   AI chat  →  files under external/
```

| Area | Role |
|------|------|
| `internal/` | Specs, CLI source (`lib/`, `cli/`) |
| `external/` | AI-written project code |
| `extensions/ilang/` | VS Code / Cursor syntax highlighting |
| `bench/` | Optional A/B measurement artifacts |

## 3. File format (`.ilang`)

### 3.1 Section syntax

- A section starts with a name on its own line, ending with `:`.
- Body lines continue until the next top-level section.
- `#` starts a line comment (useful inside generated `prompt:`).

```
section_name:
  body line
  another line
```

Section names are case-insensitive when parsed; writing them in lowercase is recommended.

### 3.2 Known sections

| Section | Required | Written by | Purpose |
|---------|----------|------------|---------|
| `lang` | yes | you | Target language for generated code |
| `context` | yes | you | Domain / stack / situation |
| `variables` | no | you | Named paths and constants |
| `goal` | yes | you | What to build |
| `constraints` | no | you | Hard limits |
| `output` | yes | you | Which files / deliverable rules |
| `prompt` | — | generated | Compressed payload for the AI — do not edit by hand |

Unknown section names are **errors**. Duplicate section names are **errors**.

### 3.3 `lang`

Must contain a `target` assignment:

```
lang:
  target = typescript
```

- Any non-empty string is accepted (no whitelist): `typescript`, `rust`, `python`, `go`, …
- Parsed into `rules.target` in the generated prompt.
- Not fed into the dictionary/`body` compression path; it is metadata for rules.

### 3.4 `variables`

Free-form `key = value` lines (convention):

```
variables:
  entry = external/main.ts
  dataFile = external/data.json
```

### 3.5 `prompt` (generated)

When you run `prompt` / `run` without `--stdout`, the CLI overwrites the `prompt:` section in the same file.

While parsing, once `prompt:` starts, **the rest of the file** is treated as its body (so nested `dict:` / `body:` / `rules:` labels do not open new top-level sections).

## 4. Prompt pipeline

### 4.1 Source text

These sections (if present and non-empty) are compressed into `body`:

- `context`
- `variables`
- `goal`
- `constraints`
- `output`

### 4.2 Dictionary

1. Tokenize with `[A-Za-z][A-Za-z0-9_-]*`.
2. Count case-insensitively; keep first-seen casing as canonical.
3. Skip English stop-words and tokens of length ≤ 1.
4. If a word appears **≥ 2** times, assign a short alias (`b`, `c`, … then `aa`, `ab`, …).
5. Replace whole-word matches in the body with aliases.

Small tasks may have little or no savings from the dictionary. Mid-size repeated vocabulary benefits more.

### 4.3 Rules block

Always injected into generated `prompt:`:

| Rule | Value | Meaning |
|------|--------|---------|
| `target` | from `lang` | Implementation language |
| `reply` | `code_only` | Prefer code, not essays |
| `no_prose` | `true` | No narrative explanations |
| `no_markdown` | `true` | No markdown wrappers |
| `no_preamble` | `true` | No intro |
| `no_postamble` | `true` | No closing summary |
| `no_comments_unless_required` | `true` | Minimal comments |
| `emit_files_under` | `external/` | Write code under this tree |

### 4.4 Generated shape

```
{
  # generated by ilang — do not edit
  rules: { ... }
  dict: { ... }
  body: {
    context: { ... }
    variables: { ... }
    goal: { ... }
    constraints: { ... }
    output: { ... }
  }
}
```

Paste this whole block into the AI chat (or print it with `--stdout`).

## 5. CLI reference

Entry: `npx tsx internal/cli/index.ts`  
Npm scripts: `npm run init` / `check` / `prompt` / `run`

Default path: `internal/index.ilang`

### 5.1 `init`

```bash
ilang init [path] [--lang <target>] [--force]
```

- Creates a starter `.ilang` file.
- Ensures `external/` exists.
- Refuses to overwrite unless `--force`.
- Picks a default `entry` extension from `lang` when possible (`ts`, `rs`, `py`, `go`, …).

### 5.2 `check`

```bash
ilang check [path]
```

Validates sections and prints `OK` or errors/warnings with hints. Exit code `1` on errors.

### 5.3 `prompt`

```bash
ilang prompt [path] [--stdout] [--quiet]
```

- Runs check first (fails if invalid).
- Default: writes `prompt:` into the `.ilang` file.
- `--stdout`: print only the prompt payload (for copy-paste). Does not require writing the file when used alone for paste workflows — still builds from current sections; with `--stdout` the CLI prints the payload and skips the “wrote file” message path for writing… Actually looking at code: with `--stdout` it only writes to stdout, does NOT update the file. Good to document.
- `--quiet`: suppress the success tip when writing to file.

### 5.4 `run`

```bash
ilang run [path] [--stdout] [--quiet]
```

Same as `prompt` after a successful check (check + generate).

### 5.5 Global flags

| Flag | Commands | Effect |
|------|----------|--------|
| `--help` / `-h` | any | Usage |
| `--lang` | `init` | Language id for template |
| `--force` | `init` | Overwrite existing file |
| `--stdout` | `prompt`, `run` | Print payload to stdout |
| `--quiet` / `-q` | `check`, `prompt`, `run` | Less console noise |

## 6. Recommended workflow

1. `npm run init -- --lang <target>`
2. Edit `context` / `goal` / `output` (and optional sections).
3. `npm run check`
4. `npm run prompt -- --stdout`
5. New AI chat → paste **only** the printed payload.
6. Verify files under `external/`.
7. Change the `.ilang` source, regenerate, new chat again.

**Do**

- Keep `output:` explicit about paths and “no explanations”.
- Use a fresh chat per serious iteration.
- Put all durable requirements in the `.ilang` file, not in chat side chatter.

**Don’t**

- Paste human sections + `prompt:` together.
- Hand-edit `prompt:`.
- Expect dictionary compression to win on tiny specs (the win is mostly `rules` / code-only behavior).

## 7. Multi-language use

ILang itself is language-agnostic. The AI target is whatever you set:

```
lang:
  target = rust
```

Align:

- `context` / `goal` wording
- file paths and extensions in `variables` / `output`
- constraints (toolchain, crates, no deps, etc.)

There is no per-language compiler backend — only `rules.target` plus your text.

- Highlights nested labels inside `prompt:` (`rules`, `dict`, `body`, …)
- Workspace may recommend installing it when you open the repo

## 9. Benchmark notes

Optional measurements live under `bench/` (see `bench/PROOF.md` if present).

Observed pattern on a mid-size multi-file task:

- Input may grow slightly (rules + dict overhead).
- **Output** tends to shrink when `code_only` / `no_prose` / `no_markdown` are respected.
- Hello-world-sized tasks are too small to demonstrate savings clearly.

Token counts in those notes use a simple chars/4 proxy unless otherwise stated.
